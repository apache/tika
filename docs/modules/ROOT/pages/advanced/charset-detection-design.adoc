= Charset Detection Pipeline
:toc:
:toc-placement: left

Tika 4.x charset detection: byte-bigram NB classifier, structural
detectors for UTF-16/UTF-32, and a script-aware text-quality arbitrator.
Design: collect candidates from every layer, then arbitrate by decode
quality — not first-detector-wins.

== Pipeline overview

The default `EncodingDetector` chain is assembled via the Java SPI.  All
results are collected into an `EncodingDetectorContext` on the
`ParseContext`; when a `MetaEncodingDetector` (currently
`JunkFilterEncodingDetector`) is present,
`CompositeEncodingDetector` switches into collect-all-then-arbitrate mode.

[cols="1,2,1,2", options="header"]
|===
| # | Detector | Module | Role

| 1
| `BOMDetector`
| `tika-core`
| Reads the first 4 bytes.  Returns a DECLARATIVE result when a
  UTF-8 / UTF-16 / UTF-32 BOM is present.

| 2
| `MetadataCharsetDetector`
| `tika-core`
| Reads `Content-Type` charset parameter and `Content-Encoding` from
  `Metadata`.  Applies WHATWG label normalization (ISO-8859-1 and
  US-ASCII → windows-1252).  DECLARATIVE.

| 3
| `MojibusterEncodingDetector`
| `tika-encoding-detector-mojibuster`
| Structural UTF-32 and UTF-16 detection, UTF-8 grammar gate, HTML
  attribute-aware stripping, then a 33-class byte-bigram NB
  classifier.  STRUCTURAL for structural hits; STATISTICAL for NB
  predictions.  See <<nb-pipeline>>.

| 4
| `HtmlEncodingDetector`
| `tika-encoding-detector-html`
| Scans HTML `<meta charset>` / `<meta http-equiv=Content-Type>` tags with a
  fast lenient regex matcher.  Returns a DECLARATIVE result.  Applies a
  curated subset of WHATWG label aliases (see <<html-charset-aliases>>).
  An alternative, spec-strict implementation — `StandardHtmlEncodingDetector`
  — is available opt-in for users who need the full WHATWG prescan algorithm.

| 5
| `JunkFilterEncodingDetector`
| `tika-ml-junkdetect`
| `MetaEncodingDetector`.  Arbitrates across all base-detector
  candidates using a script-aware text-quality model.  See <<junk-filter>>.
|===

=== EncodingResult and ResultType

Detectors return `List<EncodingResult>`.  Each carries:

* `charset` — the detected `java.nio.charset.Charset`.
* `confidence` — 0.0 to 1.0.
* `label` — the detector's internal label (may be finer than the Java
  charset name, e.g. `IBM424-ltr`, `UTF-16-BE`).
* `resultType` — one of `DECLARATIVE`, `STRUCTURAL`, `STATISTICAL`.

In the default chain, JunkFilter arbitrates across all tiers by
decode-quality scoring.  Declared charsets aren't automatically
authoritative, but a declared charset whose decoding is byte-identical
to at least one other candidate's wins.  See
<<opting-out-of-arbitration>> for plain first-match-wins.

[[nb-pipeline]]
== MojibusterEncodingDetector

The pipeline runs layers in order and returns the first confident
structural answer when one fires; otherwise it falls through to NB.

=== Layer 1 — UTF-32 structural (`WideUnicodeDetector`)

Every 4-byte group is interpreted as a 32-bit codepoint in both BE and
LE order and checked for Unicode validity (0x000000 – 0x10FFFF,
excluding surrogates).  Valid codepoints occupy only ~0.004% of the
32-bit space, so non-UTF-32 bytes almost always produce out-of-range
values within the first 8 bytes.

=== Layer 2 — UTF-16 specialist (`Utf16SpecialistEncodingDetector`)

A maxent classifier trained on stride-2 column-histogram features.
The model sees "byte distribution at even positions" vs "byte
distribution at odd positions" — the structural property that
distinguishes UTF-16 from legacy byte encodings.

An external column-asymmetry evidence gate blocks the specialist on
probes that look unambiguously single-stream (e.g. legacy CJK).  When
the specialist has a confident LE-vs-BE winner, the pipeline
short-circuits.

Byte bigrams (stride-1) cannot reliably discriminate UTF-16 of CJK
content from legacy CJK encodings, because CJK characters encode to
byte pairs that alias common ASCII byte patterns (e.g. U+6572 is
`72 65` in UTF-16-LE — same bytes as ASCII "re").  That's why UTF-16
lives outside the main NB classifier.

=== Layer 3 — UTF-8 grammar gate (`StructuralEncodingRules.checkUtf8`)

Walks the probe looking for valid UTF-8 lead + continuation byte
sequences.  Three outcomes:

* `LIKELY_UTF8` — grammar is valid and at least one complete multi-byte
  sequence is present.  Pipeline emits UTF-8 as a STRUCTURAL candidate
  alongside NB's output (safety net for short probes where NB could
  be fooled by a single coincidental bigram).
* `NOT_UTF8` — grammar is violated.  UTF-8 is filtered out of NB's
  output as a post-pass disqualifier.
* `AMBIGUOUS` — no complete multi-byte sequence (pure ASCII, or only
  a truncated lead at probe-end).  No emission.

=== Layer 4 — HTML stripping (content-type aware)

When the probe looks like HTML/XML (explicit content-type or unknown),
`HtmlByteStripper` performs a byte-level state-machine strip to leave
just content bytes for NB feature extraction.  Optimizations:

* **Script / style / comment bodies** are dropped entirely — they
  contribute no natural-language signal.
* **Text-bearing attribute values** (`alt`, `title`, `placeholder`,
  `aria-label`, `summary`, `label`) are preserved, separated by
  spaces.  These attributes carry user-visible text in the document's
  charset and provide meaningful signal on pages whose body is sparse
  (Arabic forums with navigation in `title=...`, CJK sites with
  screen-reader `aria-label=...`, etc.).
* **Tag-count backoff** — if stripping produces zero well-formed tags,
  the original bytes are used (the probe wasn't really HTML; common
  for plain text with stray `<` bytes or EBCDIC-encoded content).

=== Layer 5 — Naive Bayes byte-bigram classifier

33 classes: CJK multibyte (Big5-HKSCS, EUC-JP, GB18030, Shift_JIS,
x-EUC-TW, x-windows-949), EBCDIC family (IBM420/424-ltr/rtl, IBM500,
IBM1047), DOS OEM (IBM850/852/855/866), Cyrillic (KOI8-R, KOI8-U),
Windows single-byte (1250-1258, 874), ISO-8859-3/16, Mac (x-MacRoman,
x-mac-cyrillic), and UTF-8.

Features are **stride-1 byte bigrams** — for probe bytes `b[0..N]`,
feature value for bigram `b[i] << 8 | b[i+1]` counts how many times
it occurs.  No hashing: each bigram is directly an index into a
65 536-entry table.

==== Per-class vocabulary

Each class has its own vocabulary sized to the top bigrams covering
99.9% of its training marginal distribution.  Latin/EBCDIC classes
need ~4-9 K bigrams; CJK classes ~15-37 K.  Uniform top-K would
either waste slots on SBCS classes (noise) or starve CJK classes.

==== Per-class Laplace smoothing α

`α(c) = α_base / V(c)`.  Peaky Latin classes get tight smoothing
(observed frequencies are trustworthy); diffuse CJK classes get
flatter smoothing (no single bigram is strong evidence).

==== Shared reference denominator

The Laplace denominator uses `N_ref = max(total_c)` across classes,
not per-class totals.  Without this, classes with smaller training
corpora (Mac variants trained on 10M bigrams vs 30M for the main
classes) end up with milder unseen-log-probabilities, giving them a
systematic free-lift on sparse-evidence probes.  Shared `N_ref`
eliminates the disparity while preserving relative within-class
frequencies.

==== IDF reweighting

A global `float[65_536]` IDF table multiplies each bigram's contribution
to every class score:
`idf[b] = log((C + 1) / (df[b] + 1))` where `df[b]` is the number of
classes in which bigram `b` has non-zero training count.  Bigrams
appearing in every class (common ASCII) contribute near zero;
class-specific bigrams (CJK signal bytes, EBCDIC-unique patterns)
carry full weight.  Bigrams with IDF quantized to zero skip the
inner class loop entirely.

==== Int8 quantization

Weights and IDF are int8-quantized with per-class scales.  Inner loop
is pure integer multiply-accumulate; a single per-class dequant
multiplication happens at end-of-probe.  Memory: 2 MB on disk (NBB3
format), ~2 MB in memory.  Accuracy: byte-exact-match with float at
every probe length we've measured.

==== Post-NB fallbacks

* **Empty / near-empty probes (< 2 bytes)** → windows-1252 @ 0.1
  confidence.  WHATWG default; never returns empty result.
* **Pure ASCII probes** (no bytes ≥ 0x80, no nulls) → windows-1252.
  Bigram NB cannot discriminate Latin code pages on pure-ASCII
  content; return the HTML5-canonical answer directly.
* **Latin-sibling → windows-1252 rewrite** — on low-evidence probes
  (< 5 high bytes), if the top NB candidate is a non-1252 member of
  the Latin family and the probe decodes byte-identically under
  windows-1252, rewrite to windows-1252.
* **MIN_EMIT_CONFIDENCE = 0.01** — noise candidates below this
  threshold are not emitted into the pool.  Prevents JunkFilter from
  scoring weak coincidence picks against NB's confident top.

[[junk-filter]]
== JunkFilterEncodingDetector — text-quality arbitration

`JunkFilterEncodingDetector` is a `MetaEncodingDetector` backed by
`JunkDetector`, a script-aware text-quality scorer that distinguishes
clean text from mojibake / wrong-codec / corrupted decodings.  Its
presence switches the composite into collect-all mode.

[IMPORTANT]
====
*JunkFilter arbitrates over ALL candidate tiers, including
DECLARATIVE.*  Real-world web declarations are unreliable, so a
`<meta charset>` tag is one input among several.

A declared charset wins when its decoding is byte-identical to at
least one other candidate's — text quality can't distinguish the two,
so honouring the declaration ensures downstream divergent bytes
(e.g. 0xA4 = € in ISO-8859-15, ¤ in windows-1252) are decoded as the
author intended.

For plain first-match-wins, omit JunkFilter (see <<opting-out-of-arbitration>>).
====

=== Arbitration outline

. **One unique candidate** — abstain; composite default ordering wins.
. **HTML byte-stripping** — `HtmlByteStripper` removes well-formed
  HTML/XML markup before decoding so the score reflects body text, not
  tags.  Falls back to the raw probe if no tags are detected.
. **Decode under each candidate** — insertion-ordered for deterministic
  tournament seeding.
. **Declarative-equivalent-decode preference** — if a DECLARATIVE
  candidate's decoded output equals at least one other candidate's, it
  wins immediately.
. **All decodings identical, no DECLARATIVE** — abstain.
. **Pairwise tournament** — first candidate seeds champion; each
  challenger compared via `JunkDetector.compare`; higher z-score wins.

=== JunkDetector scoring

`JunkDetector` partitions decoded text into maximal Unicode-script runs
(COMMON / INHERITED / UNKNOWN attach to the preceding run).  Each run
is scored against its own per-script model (byte-bigram log-prob table
+ named-Unicode-block transitions + control-byte fraction) into a
per-run logit.  Logits are combined into an overall z-score weighted by
run byte count, so mixed-script text is scored fairly.

Scripts not in the trained model are treated as neutral (z = 0), not
junk — this prevents a garbled-but-recognisable decoding from beating
a correct decoding whose script is unknown to the model.

=== Why HTML stripping is essential

The score is a byte-weighted average over script runs.  An unstripped
HTML probe with 10 KB of whitespace + tags and 24 bytes of body gives
the body a weight of ~0.0024 — invisible.  Stripping gets the probe to
a content-only byte budget where scripts compete on comparable footing.

`HtmlByteStripper` operates on raw bytes (ASCII tag delimiters survive
every charset Tika cares about).  `<script>`, `<style>`, and comment
bodies are dropped; text-bearing attribute values (`alt`, `title`,
`placeholder`, `aria-label`, `summary`, `label`) are preserved.

[[opting-out-of-arbitration]]
=== Opting out — first-match-wins

This is *not* the default chain — it's a custom configuration.
By default, JunkFilter arbitrates across all candidates as described
above.  To opt out, omit `JunkFilterEncodingDetector` from the chain;
the composite detector then runs in plain first-match-wins mode (each
base detector's top result is taken in registration order):

[source,json]
----
{
  "encoding-detectors": [
    { "bom-detector": {} },
    { "metadata-charset-detector": {} },
    { "standard-html-encoding-detector": {} },
    { "mojibuster-encoding-detector": {} }
  ]
}
----

In this opt-out chain, BOM / metadata / meta-tag declarations are
honoured authoritatively when present, and NB only runs when no
declaration fires.  Trade-off: lying declarations propagate
unfiltered, and Mojibuster's top statistical guess is taken without
any text-quality cross-check.

== Performance

Evaluated on a held-out 37-charset devtest corpus (~1.22M probes per
probe length).  Raw outputs are preserved in
`tika-encoding-detectors/tika-encoding-detector-mojibuster/docs/performance/`.

The default chain trades occasional overrides of correct declarations
on short probes for resilience against lying declarations and wrong-codec
decodings.

== MetadataCharsetDetector

A lightweight detector in `tika-core` that reads declarative charset hints from
the `Metadata` object before any byte analysis:

* `Content-Type` charset parameter (e.g. `text/html; charset=windows-1251`)
* `Content-Encoding` (used by `RFC822Parser` and similar MIME-aware parsers)

Applies WHATWG label normalization: `ISO-8859-1` and `US-ASCII` are mapped to
`windows-1252` because browsers (and the HTML5 spec) treat them as aliases for
windows-1252 in practice.

Returns a DECLARATIVE result, which `JunkFilterEncodingDetector` honours when
its decoding is byte-identical to at least one other candidate's.

== BOMDetector

Reads the first 4 bytes and detects:

[cols="1,1", options="header"]
|===
| Byte sequence | Encoding

| `EF BB BF` | UTF-8
| `FF FE 00 00` | UTF-32-LE
| `00 00 FE FF` | UTF-32-BE
| `FF FE` | UTF-16-LE
| `FE FF` | UTF-16-BE
|===

Returns a DECLARATIVE result.  The HTML detectors do not handle BOMs on their
own: `BOMDetector` is the sole source of BOM evidence, which lets
`JunkFilterEncodingDetector` arbitrate when a BOM and a `<meta charset>` tag
disagree.

== Performance and accuracy

Numbers are from the held-out MADLAD-400 + Wikipedia devtest set (model v6).
1 469 647 samples across 41 charsets including structural-only charsets
(US-ASCII, ISO-2022-JP/KR/CN, UTF-32-BE/LE).

"All" = ML model + structural pre-filters + all post-processing rules — this
is the production configuration.  UTF-32 is detected by a structural
pre-filter (validity-only codepoint check inspired by ICU4J's
`CharsetRecog_UTF_32`); UTF-16 uses a combination of structural phases
(null-column, low-block-prefix) and the ML model for CJK cases.  The ML model
has 35 classes (UTF-32 labels are excluded from training since they are handled
structurally).

Raw eval output: `eval-v6-no-utf32.txt` in this directory.

=== Overall accuracy

[cols="2,1,1,1", options="header"]
|===
| Metric | Mojibuster (All) | ICU4J | juniversalchardet

| Strict accuracy (full probe) | **95.0%** | 45.5% | 33.5%
| Soft accuracy (full probe)   | **97.3%** | 67.0% | 42.0%
| Decode-match (full probe)    | **99.4%** | 54.8% | 41.2%
| Alpha-match (full probe)     | **99.8%** | 55.3% | 43.5%
| Latency (full probe)         | **~13 µs** | ~147 µs | ~16 µs
|===

**Strict accuracy** = exact charset name match. +
**Soft accuracy** = exact or confusable-group match (e.g. predicting IBM500 for
an IBM1047 file counts as soft-correct since they share 247 of 256 byte
mappings). +
**Decode-match** = the predicted charset decodes the probe bytes to the same
string as the true charset (the prediction is functionally correct even if the
label differs). +
**Alpha-match** = same as decode-match but ignoring non-alphanumeric characters.

== Training

Training tools live in `org.apache.tika.ml.chardetect.tools`.  Core
flow:

[source,bash]
----
# 1. Train (bigram Naive Bayes, per-class V99.9% vocab, int8 quantized)
java -cp tika-ml/tika-ml-chardetect/target/tika-ml-chardetect-*-tools.jar \
  org.apache.tika.ml.chardetect.tools.TrainNaiveBayesBigram \
  --data ~/datasets/charset-detect/train \
  --output nb-bigram.bin \
  --coverage 0.999 \
  --alpha-base 1.0 \
  --max-samples-per-class 50000

# 2. Install model into the mojibuster module resources
cp nb-bigram.bin \
  tika-encoding-detectors/tika-encoding-detector-mojibuster/src/main/resources/\
org/apache/tika/ml/chardetect/nb-bigram.bin

# 3. Rebuild
mvn install -pl tika-encoding-detectors/tika-encoding-detector-mojibuster \
  -DskipTests

# 4. Evaluate against ICU4J and juniversalchardet on devtest
java -cp tika-ml/tika-ml-chardetect/target/tika-ml-chardetect-*-tools.jar \
  org.apache.tika.ml.chardetect.tools.EvalCharsetDetectors \
  --nb-model nb-bigram.bin \
  --data ~/datasets/charset-detect/devtest \
  --lengths 20,100,256,full
----

Training data is sourced from MADLAD-400 (multilingual web text) and
Wikipedia dumps, encoded to each target charset with charset-specific
normalization (Arabic Yeh forms for windows-1256, Hebrew nikkud
stripping for IBM424, punctuation mapping for legacy charsets that
can't encode typographic characters).

== On-disk model format

`.bin` files use magic `NBB3` and store:

* IDF scale + quantized IDF table (65 536 bytes).
* Per class: label, dequantization scale, quantized unseen-floor
  value, vocabulary size, and each trained bigram as
  `(uint16 bigram, int8 logP)` pairs.

Files for the shipped 33-class model are ~1 MB on disk.  Loader
materializes a dense `logP8[65 536 × numClasses]` array filled with
per-class unseen floors, overwritten by trained pairs.  Working-set
memory: ~2 MB.

== Why Naive Bayes

Earlier Tika versions used multinomial logistic regression (maxent)
with feature hashing.  The move to NB was driven by:

* **Speed**: direct bigram indexing removes the hash + bucket-lookup
  cost.  Inner loop is `score[c] += logP[b × numClasses + c] × idf[b]`
  with no branching (zero-IDF bigrams are skipped before the class
  loop).  Measured ~15 µs on a full 1 KB probe for 33 classes.
* **Memory layout**: bigram-major byte arrays fit in L3 cache for the
  full table.  Sequential access through the hot loop is cache-line
  efficient.
* **Interpretability**: per-class training totals, vocabulary sizes,
  and per-class α are all visible in the saved model.  Makes
  calibration issues (class bias from training size, smoothing spread,
  etc.) diagnosable.
