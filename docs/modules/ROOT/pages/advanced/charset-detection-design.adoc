= Charset Detection: Design Decisions
:toc:
:toc-placement: left

Working document tracking design decisions for the ML-based charset detection
pipeline (`tika-ml-chardetect`). Updated as decisions are made.

== Goals

Replace the existing ICU4J and Universal (juniversalchardet) heuristic detectors
with a trainable ML model that generalises across more encodings and languages,
while preserving the existing `CharSoupEncodingDetector` arbitration mechanism
and the `EncodingDetector` chain architecture.

== Evaluation Results

The ML detector was evaluated against the ICU4J and juniversalchardet baselines
using `EvalCharsetDetectors` on the MadLAD-derived test corpus.
Results justified removing ICU4J and Universal from the default detection chain.

== Architecture

module at the project root (peer to `tika-parsers`, `tika-detectors`, etc.):

|===

| `tika-core`
| `WideUnicodeDetector`, `BOMDetector`

| No extra dependencies; always available

| `tika-encoding-detector-ml`
| `MojibusterEncodingDetector`
| yes (default)
| ML byte-ngram model; replaces ICU4J + Universal

| `tika-encoding-detector-html`
| `HtmlEncodingDetector`, `StandardHtmlEncodingDetector`

| `CharSoupEncodingDetector`

| `tika-encoding-detector-icu4j`
| `Icu4jEncodingDetector` + vendored ICU4J charset engine

| Kept for compatibility; not in default chain

=== Detection chain

Default detectors run in order; all results are collected into `EncodingDetectorContext`:

. `BOMDetector` ‚Äî definitive (confidence 1.0) when a BOM is present (opt-in, not default)
. `WideUnicodeDetector` ‚Äî UTF-16/32 structural detection; writes `WideUnicodeSignal`
  into `EncodingDetectorContext` (see <<wide-unicode-signal>>)
. `MojibusterEncodingDetector` ‚Äî ML-based detector (replaces ICU4J + Universal)
. `HtmlEncodingDetector` ‚Äî charset from HTML `<meta>` tags
. `CharSoupEncodingDetector` ‚Äî meta-detector; arbitrates using language scoring

=== EncodingResult

Detectors now return `List<EncodingResult>` instead of a single `Charset`.
Each `EncodingResult` carries a `charset` and a `confidence` (0.0‚Äì1.0).
This mirrors the language detection API and enables arbitration across multiple
candidates.

=== EncodingDetectorContext

Created by `CompositeEncodingDetector` for each detection pass, stored in
`ParseContext`, and *removed after detection completes* so it does not bleed
into embedded-document parses.

Contains:

* Per-detector ranked `List<EncodingResult>` contributions
* `WideUnicodeSignal` written by `WideUnicodeDetector` (see below)
* Arbitration metadata set by `CharSoupEncodingDetector`

[[wide-unicode-signal]]
=== WideUnicodeSignal

`WideUnicodeDetector` writes a `WideUnicodeSignal` into `EncodingDetectorContext`
(not directly into `ParseContext`) when it finds wide-Unicode structure.
The signal carries:

* `Kind.VALID` + detected `Charset` ‚Äî structurally valid UTF-16/32
* `Kind.ILLEGAL_SURROGATES` ‚Äî null-column pattern present but surrogate pairs are invalid

`MojibusterEncodingDetector` checks for this signal before touching the probe bytes.
If present it returns an empty list immediately ‚Äî the model was not trained on
null-heavy UTF-16/32 data and would produce noise.

Using `EncodingDetectorContext` (rather than `ParseContext`) as the carrier
ensures the signal is scoped to the current detection pass and cannot affect
embedded-document detection.

=== ISO-8859-X ‚Üí Windows-12XX post-correction

**Decision:** rule-based post-correction, not training data.

After the model runs and group probabilities are collapsed, if the probe
contains any byte in the C1 range (`0x80‚Äì0x9F`):

* Those bytes are C1 control codes in every ISO-8859-X standard
* They are printable characters in every Windows-12XX encoding

Therefore, any ISO-8859-X result is structurally invalid when C1 bytes are
present.  `upgradeIsoToWindows()` replaces each such result with its
Windows-12XX equivalent from `CharsetConfusables.ISO_TO_WINDOWS`, preserving
the model's confidence score.

The upgrade map covers all confusable ISO/Windows pairs:
`ISO-8859-1` / `ISO-8859-15` ‚Üí `windows-1252`, `ISO-8859-2` ‚Üí `windows-1250`,
`ISO-8859-5` ‚Üí `windows-1251`, `ISO-8859-6` ‚Üí `windows-1256`,
`ISO-8859-7` ‚Üí `windows-1253`, `ISO-8859-8` ‚Üí `windows-1255`,
`ISO-8859-9` ‚Üí `windows-1254`, `ISO-8859-4` / `ISO-8859-13` ‚Üí `windows-1257`.

This rule is preferred over generating split training data because:

* The 0x80‚Äì0x9F signal is sparse in real text (smart quotes, Euro sign appear
  in a minority of documents); the model may not learn reliable weights
* The rule is deterministic and covers all confusable groups simultaneously
* The model's script-level discrimination (Western European vs. Cyrillic vs.
  Arabic etc.) is trusted; this only fixes ISO-vs-Windows within the group

=== Confusable groups

`CharsetConfusables` defines symmetric groups (both directions acceptable for
evaluation) and directional superset chains (predicting a superset is safe;
predicting a subset is not).

`collapseGroups()` pools softmax probability within each group and awards the
total to the highest-scoring member before thresholding.

== CJK encoding detection (replacing ICU4J)

=== Problem

The byte-ngram ML model is trained on single-byte and UTF-8 content.
For CJK multi-byte encodings (Shift_JIS, EUC-JP, EUC-KR, Big5, GB18030) it
provides useful discrimination on long probes (‚â• ~50 high bytes) but is
unreliable on short probes such as ZIP entry filenames (10‚Äì30 bytes).

With ICU4J removed there is no backstop for these short probes.

=== Solution: grammar walkers + gated invocation

Implement grammar walkers for each CJK encoding, inspired by ICU4J's
`CharsetRecog_mbcs` class.  Each walker:

. Iterates the probe byte-by-byte according to the encoding's grammar rules
. Counts valid double-byte characters, single-byte characters, and errors
. Returns a confidence score

Invocation is *gated on the model's top-N results*: grammar walkers only run
when a CJK encoding appears in the model output.  This avoids false positives
on Latin/Arabic/Cyrillic probes (where some byte values coincidentally match
CJK lead-byte ranges) and avoids wasting cycles on clearly non-CJK content.

Grammar walker complexity (lines of logic, excluding data):

|===
| Encoding | Walker lines | Notes

| Shift_JIS | ~10 | Lead 0x81‚Äì0x9F / 0xE0‚Äì0xFC + trail 0x40‚Äì0x7E / 0x80‚Äì0xFF; half-width katakana 0xA1‚Äì0xDF single byte
| EUC-JP / EUC-KR | ~35 (shared) | ‚â§0x8D single; 0xA1‚Äì0xFE two-byte; 0x8E SS2; 0x8F SS3 three-byte
| Big5 | ~10 | Lead 0x81‚Äì0xFE; trail 0x40‚Äì0xFE excluding 0x7F / 0xFF
| GB18030 | ~30 | Two-byte or four-byte (second 0x30‚Äì0x39); most complex
| Confidence formula | ~15 | Shared; based on doubleByteCount, badCharCount, commonCharCount
|===

Total: ~100 lines of logic across all five encodings.

=== Common character frequency tables

**Decision:** derive from corpus later; start with grammar-walker-only.

ICU4J uses hardcoded frequency tables (64 entries each) of the most common
two-byte characters per encoding.  These boost confidence on short probes via
a logarithmic formula.

Without the tables, grammar-walker-only confidence for a 10-byte valid SJIS
probe is ~33/100 ‚Äî enough for discrimination (ruling out wrong encodings) but
below the typical acceptance threshold for a positive result.

Plan:

. **Now:** implement grammar walkers only.  Discrimination between CJK
  encodings works correctly.  Short-probe positive confidence may be low.
. **Later, if short-probe confidence proves insufficient:** derive common
  character tables from MadLAD corpus:
  ** Take Japanese/Chinese/Korean text, re-encode to target encoding
  ** Count frequency of every two-byte sequence
  ** Take the top 64, sorted ‚Äî drop in as the `commonChars` array
  ** This produces corpus-specific tables tuned to our training distribution

== Feature set decision ‚Äî UB-AS, 16384 buckets

Selected via grid search over `charset-detect3` corpus (34 charsets, 20K train /
2K devtest samples each, MadLAD source).  Each config trained 3√ó and averaged
to reduce SGD noise.

[cols="1,1,1", options="header"]
|===
| Config | Buckets | Avg devtest accuracy

| UB-AS | 32768 | 95.28%
| UB-AS | 16384 | 95.27%
| UB--S | 16384 | 95.26%
| UB-AS |  8192 | 95.24%
| UB--S | 32768 | 95.21%
| UB--S |  8192 | 94.80%
|===

**Decision: UB-AS at 16384 buckets.**

Rationale:

* Trigrams (T) were dropped ‚Äî no accuracy improvement over UB-AS, adds hash
  collisions and model size.
* Anchored bigrams (A) add ~0.04% consistently across bucket sizes.  Small but
  reproducible, and conceptually sound (captures Shift-JIS/Big5 low trail bytes
  at character boundaries).
* Accuracy plateau from 8192 to 32768 buckets is within SGD noise (~0.1%).
  16384 chosen as the best size/accuracy trade-off (half the model vs. 32768).
* Stride-2 bigrams (S) are the largest single accuracy driver ‚Äî lifting overall
  accuracy from ~73% (UBT- without UTF-16/32 training) to ~95% by giving the
  model direct code-unit visibility into UTF-16/32 structure.

=== UTF-16/32 training change

UTF-16-LE/BE and UTF-32-LE/BE were previously excluded from ML training
(`STRUCTURAL_ONLY_CHARSETS`) and detected solely by `WideUnicodeDetector`
structural heuristics.  They are now trained in the ML model:

* Added to `UNICODE_CHARSETS` ‚Äî samples generated from **all** MADLAD languages
  (Arabic, Hebrew, Greek, Japanese, Korean, Thai, Chinese, Russian, etc.) using
  BOM-free codecs (`utf-16-le`, `utf-16-be`, `utf-32-le`, `utf-32-be`).
* U+FEFF stripped from all source text before encoding ‚Äî the model never sees
  BOM bytes.  `BOMDetector` owns that signal at inference time.
* Stride-2 bigrams give the model structural visibility: UTF-16LE ‚Üí many
  `(XX, 0x00)` pairs; UTF-16BE ‚Üí `(0x00, XX)`; UTF-32 ‚Üí dense `(0x00, 0x00)`.

== Training data plan

=== Single-byte encodings

Source: MadLAD (cleaned UTF-8).  Re-encode each document into the target
single-byte encoding.

Bucketing rules for Western European encodings to ensure the model can
distinguish ISO-8859-1 from windows-1252:

* **Text with characters exclusive to windows-1252** (U+20AC ‚Ç¨, U+2018/2019
  smart quotes, U+201C/201D, U+2013 en-dash, U+2014 em-dash, etc.) can only
  be re-encoded as windows-1252.  Generate windows-1252 examples only.
  These examples will have bytes in 0x80‚Äì0x9F, training the model to associate
  those bytes with windows-1252.
* **Text with only characters in the shared 0xA0‚Äì0xFF range** is byte-for-byte
  identical in ISO-8859-1 and windows-1252.  Label as ISO-8859-1 as the
  canonical default.  These are genuinely ambiguous; the post-correction rule
  handles the case where such a file actually has 0x80‚Äì0x9F bytes at runtime.
* **Pure ASCII text** ‚Üí UTF-8 / US-ASCII; excluded from this group.

The same bucketing applies to each ISO/Windows confusable pair
(ISO-8859-2 / windows-1250 for Central European, etc.).

=== CJK encodings

Re-encode MadLAD Japanese/Chinese/Korean text into Shift_JIS, EUC-JP, EUC-KR,
Big5, GB18030.  Apply OOV-rate filter: exclude chunks where fewer than 20% of
bytes are high bytes (too ASCII-heavy to provide useful CJK signal).

== Removed from the default chain

* `ByteEncodingHint` ‚Äî removed; replaced by `WideUnicodeDetector`
* `Icu4jEncodingDetector` ‚Äî removed from default SPI chain; still available as
  `tika-encoding-detector-icu4j` for explicit opt-in configuration
* `UniversalEncodingDetector` ‚Äî removed from default SPI chain; still available as
  `tika-encoding-detector-universal` for explicit opt-in configuration
* `BOMDetector` ‚Äî moved from `tika-parser-text-module` to `tika-core`
  (package `org.apache.tika.detect`); remains opt-in (not SPI-loaded by default)
